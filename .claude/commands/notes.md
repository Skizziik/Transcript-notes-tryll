Ты получил команду на транскрипцию аудио и создание детальных заметок.

## Аргументы

`$ARGUMENTS`

Поддерживается только опциональный флаг `--lang XX` для принудительного указания языка. Пути передавать **нельзя** — источник всегда `input/`.

## Шаг 0 — найти аудио в input/

```bash
shopt -s nullglob 2>/dev/null
ls -1 input/ 2>/dev/null | grep -iE '\.(m4a|mp3|wav|mp4|mov|opus|aac|flac|aiff|ogg|webm)$' || true
```

Правила:
- **0 файлов** → скажи пользователю: «Положи **один** аудиофайл в `input/` и запусти `/notes` снова» и **остановись**.
- **>1 файла** → перечисли их и скажи: «В `input/` должен быть ровно один аудиофайл. Убери лишние и запусти снова». **Остановись.**
- **1 файл** → продолжай.

Пусть `<audio>` — это полный путь к найденному файлу, а `<base>` — имя без расширения.

## Шаг 1 — подготовить папку под запуск

Каждый запуск пишет в собственную папку, чтобы старые результаты не перетирались.

```bash
ts=$(date +%Y-%m-%d_%H-%M)
outdir="output/${base}_${ts}"
mkdir -p "$outdir"
```

Все артефакты этого запуска кладём именно в `$outdir`.

## Шаг 2 — транскрипция

```bash
.venv/bin/python scripts/transcribe.py "<audio>" --out "$outdir"
```

Если в аргументах был `--lang XX` — добавь `--language XX`.

Транскрипция занимает время (≈0.3–1× от длительности аудио; первый запуск качает модель ~3 GB). **Не ставь короткий timeout** — для часовой записи может занять 15–30 минут. Жди завершения.

После: прочитай `$outdir/<base>.txt` и `$outdir/<base>.json` (сегменты с таймкодами — пригодятся на следующем шаге). Сообщи пользователю язык, длительность (из `<base>.json`), длину транскрипта в символах.

## Шаг 3 — заметки

Сохрани в `$outdir/<base>.notes.md`. Принципы:

- **Основной язык — русский.** Английские термины, имена, инструменты — оставляй как есть (`GitHub`, не «ГитХаб»).
- **Структура** — иерархия `##` / `###` по темам. В начале короткий блок «О чём» (3–5 пунктов).
- **Детализация, не саммари.** Сохраняй цифры, имена, команды, примеры, цитаты. Определения терминов оформляй явно: **GraphQL** — query language for APIs…
- **Решения и Action Items** — отдельный раздел, по пунктам, с ответственными и сроками если звучали.
- **Открытые вопросы** — отдельный раздел если что-то осталось неясно/отложено.
- **Сильные цитаты** — `> блок`, по возможности с указанием говорящего.
- **Код/команды** — в ` ``` ` блоках.
- **Таймкоды `[MM:SS]`** — если запись длиннее ~10 минут, ставь рядом с `###` заголовками подтем, ключевыми action items и цитатами, чтобы из `.docx` можно было прыгнуть к моменту записи. Бери `start` ближайшего сегмента из `<base>.json`, где тема впервые звучит, и форматируй как `[MM:SS]` (для записей длиннее часа — `[H:MM:SS]`). Не сыпь таймкоды на каждом пункте — только там, где они реально помогают навигации. Если запись короче ~10 минут — таймкоды не нужны, лишний шум.
- **Не выдумывай.** Плохо расслышанное — `[неразборчиво]` или `[?]`. Если не уверен в таймкоде — не ставь.
- **Не добавляй** «Выводы»/«Заключение», если их не было в исходнике.

Длина — пропорциональна записи (получасовая встреча → 1–2 страницы плотных заметок).

## Шаг 4 — конвертация в .docx

```bash
.venv/bin/python scripts/to_docx.py "$outdir/<base>.notes.md" -o "$outdir/<base>.notes.docx"
```

## Шаг 5 — отчёт пользователю

Сообщи пользователю **путь к папке `$outdir`** и кратко её содержимое:

```
output/<base>_YYYY-MM-DD_HH-MM/
  <base>.txt            — сырой транскрипт
  <base>.json           — сегменты с таймкодами
  <base>.notes.md       — заметки (можно править)
  <base>.notes.docx     — финальный документ ← это главное
```

И 2–3 пункта о чём запись, чтобы видно было что обработалось правильно.

Всё. Только пайплайн.
